# blind-ml — Train ML models on encrypted data

[![Watch Demo Recording](https://i9.ytimg.com/vi/N9VNa7xC_48/mqdefault.jpg?sqp=CIDA5s0G-oaymwEmCMACELQB8quKqQMa8AEB-AH-CYAC0AWKAgwIABABGD8gEyh_MA8=&rs=AOn4CLDpMxVDSvv8_M38H_ROE8uBJrjTAg)](https://www.youtube.com/watch?v=N9VNa7xC_48 "Encrypted Fraud Training")

**Train sklearn-style machine learning models on sensitive data that never gets decrypted, via [Blind Insight](https://blindinsight.com)'s searchable encryption.**

This repo ships **two production-ready demo notebooks**:

| Notebook | Domain | Models | Records |
|----------|--------|--------|---------|
| [`fraud.ipynb`](fraud.ipynb) | Cross-border financial fraud (IBANs, jurisdictions, fraud reports) | Naive Bayes + Decision Tree + Logistic Regression | up to 600K |
| [`breast_cancer.ipynb`](breast_cancer.ipynb) | Healthcare risk prediction (Gail-model risk factors, HIPAA k=11 binning) | Decision Tree | up to 100K |

Both demos achieve F1 parity with their sklearn plaintext counterparts while training **only on encrypted aggregate queries** — no record-level decryption.

The walkthrough below uses the fraud demo as the primary example. Steps for the breast-cancer demo are analogous; see the [Breast-cancer demo](#breast-cancer-demo) section for the few differences.

---

## Why this matters

This notebook demonstrates how banks and financial institutions can build fraud detection systems while keeping sensitive data (IBANs, jurisdictions, fraud reports) fully encrypted - enabling cross-border collaboration without exposing raw data.

## What Does This Demo Show?

| Traditional ML | Blind Insight ML |
|----------------|------------------|
| Data must be decrypted | Data stays **encrypted** |
| Exposes raw records | Only aggregate statistics returned |
| Privacy/compliance risk | GDPR, DORA, CCPA compliant |
| Centralized data silos | Secure cross-organization sharing |

**The Result:** All three encrypted models — Naive Bayes, Decision Tree (CART/Gini), and Logistic Regression (OLS+IRLS) — achieve **identical F1 scores** to their sklearn plaintext counterparts (F1=0.942, 0pp gap). Training uses **~90 aggregate queries on 600K encrypted records** — no record-level decryption needed.

**Data integrity note:** The local SQLite files (`demo_data/plaintext/fraud_train.db`, `fraud_test.db`) are generated by `scripts/generate_fraud_data.py` and match the JSON upload batches record-for-record. BI is the source of truth for encrypted data; the local DBs exist for side-by-side plaintext comparison. All plaintext values respect BI schema min/max; for integer fields, the schema applies `maximum = actual_max + 2` padding.

**Flexible scale:** Each 50K training batch has the same feature distributions, so the demo works at any scale from 50K to 500K in 50K increments. Upload as many batches as your account allows.

### Create dataset and schemas first (Blind Insight)

Create a dataset and schemas in Blind Insight using the Blind CLI and the included schema file. Choose any names you like and set the slugs in your `.env`.

1. **Create the dataset** (use your organization slug from `./blind organization list`):
   ```bash
   ./blind dataset create --organization YOUR_ORG --name "Fraud Data" --description "Fraud demo"
   ```
   The CLI generates a slug from the name (e.g. `fraud-data`). Note it for the next step.

2. **Create the train and test schemas** (same schema definition for both). Use the dataset slug from the previous step. Per [official docs](https://docs.blindinsight.io/getting-started/using-the-blind-proxy/#create-a-schema):
   ```bash
   ./blind schema create --name Train --dataset YOUR_DATASET_SLUG --organization YOUR_ORG --description "Fraud training records" --file schemas/fraud.json
   ./blind schema create --name Test --dataset YOUR_DATASET_SLUG --organization YOUR_ORG --description "Fraud test records" --file schemas/fraud.json
   ```
   The slug is derived from the name (e.g. `train`, `test`). The file `schemas/fraud.json` in this repo matches the demo data.

3. Note your schema URLs from the output (the `"url"` field). For example:
   ```
   https://api.app.blindinsight.io/api/schemas/YOUR_SCHEMA_ID/
                                                ^^^^^^^^^^^^^^^
                                                this is the schema ID
   ```
   You can also find them later with `blind schema list`. You'll need the full URL for the upload steps below.

### Uploading the demo data to Blind Insight

The repo includes **600K training** (500K base + 100K noise) and **54K test** (50K base + 4K noise) records in two forms:

- **Plaintext (SQLite):** `demo_data/plaintext/fraud_train.db` and `fraud_test.db` — used by the notebook for local model comparison.
- **Upload batches:** `demo_data/upload_batches/` — JSON files in Blind Insight upload format. Each training batch contains 50K records; the test batch contains 50K records. Run `python3 scripts/generate_fraud_data.py --append-noise` after the initial generation to add the noise records to the SQLite databases.

You don't have to upload all 10 training batches. Each batch has the same feature distributions, so the demo works with 1 batch (50K), 5 batches (250K), or all 10 (500K).

**Before uploading:** Replace the schema URL placeholder in the batch files with your actual schema IDs (use the relative URL format per [official docs](https://docs.blindinsight.io/getting-started/uploading-data/#json-support)):

```bash
# Replace in all train batch files (use schema ID from `blind schema list`):
sed -i '' 's|REPLACE_WITH_YOUR_SCHEMA_URL|https://api.app.blindinsight.io/api/schemas/YOUR_TRAIN_SCHEMA_ID/|g' demo_data/upload_batches/fraud_train_batch_*.json

# Replace in the test batch file:
sed -i '' 's|REPLACE_WITH_YOUR_SCHEMA_URL|https://api.app.blindinsight.io/api/schemas/YOUR_TEST_SCHEMA_ID/|g' demo_data/upload_batches/fraud_test_batch_01.json
```

There are three ways to upload. See the [Uploading Data docs](https://docs.blindinsight.io/getting-started/uploading-data/) for full details.

#### Method 1: Web UI (drag and drop) — Easiest

1. Open the Blind Insight web app (behind the proxy at `https://local.blindinsight.io`) and navigate to your dataset.
2. Select your **train** schema and click into the records view.
3. Drag and drop a `fraud_train_batch_*.json` file onto the upload area. The proxy encrypts records client-side using your keyring before sending to the server.
4. A progress bar shows upload status. Wait for it to complete before uploading the next batch.
5. Repeat for as many training batches as you want (1-10).
6. Switch to your **test** schema and drag-and-drop `fraud_test_batch_01.json`.

**Tip:** When using the Web UI, the `schema` field in records is auto-detected — you can skip the `sed` step above.

See [Web UI docs](https://docs.blindinsight.io/getting-started/uploading-data/#web-ui) and [Chrome Extension docs](https://docs.blindinsight.io/getting-started/using-the-chrome-extension/) for setup.

#### Method 2: Proxy REST API (curl) — Best for Bulk CLI Uploads

Make sure the proxy is running (`./blind proxy`) in another terminal. Post directly to the proxy's `/api/jobs/upload/` endpoint with HTTP Basic auth:

```bash
# Upload training batches (10 files x 50K each = 500K)
for f in demo_data/upload_batches/fraud_train_batch_*.json; do
  echo "Uploading $f..."
  JOB_ID=$(curl -s -X POST 'https://local.blindinsight.io/api/jobs/upload/' \
    -u "$BI_EMAIL:$BI_PASSWORD" \
    -H 'Content-Type: application/json' \
    --data-binary "@$f" | python3 -c "import sys,json; print(json.load(sys.stdin)['job_id'])")
  echo "  Job: $JOB_ID — polling status..."
  while true; do
    sleep 10
    STATUS=$(curl -s "https://local.blindinsight.io/api/jobs/$JOB_ID/" \
      -u "$BI_EMAIL:$BI_PASSWORD")
    echo "  $STATUS"
    echo "$STATUS" | grep -q '"complete"' && break
    echo "$STATUS" | grep -q '"failed"' && echo "FAILED!" && exit 1
  done
  echo "  Done."
  sleep 2
done

# Upload test batch (1 file x 50K)
curl -X POST 'https://local.blindinsight.io/api/jobs/upload/' \
  -u "$BI_EMAIL:$BI_PASSWORD" \
  -H 'Content-Type: application/json' \
  --data-binary '@demo_data/upload_batches/fraud_test_batch_01.json'
```

This method posts JSON directly to the proxy, which encrypts and forwards to the cloud. It returns a `job_id` you can poll for progress.

#### Method 3: CLI (`blind jobs upload`)

```bash
blind jobs upload --data demo_data/upload_batches/fraud_train_batch_01.json
```

**Note:** In our testing (2026-03), the CLI `--host` / `-H` flag did not reliably route uploads through the local proxy — jobs were accepted but records didn't land. If you experience this, use Method 2 (Proxy REST API) instead.

You can upload fewer training batches if you prefer (e.g. just `fraud_train_batch_01.json` for a quick 50K test).

See `demo_data/README.md` for a quick reference on the upload steps.

## What's Happening Under the Hood?

### The Business Problem

A bank needs to decide whether to approve a new account or issue a loan. The fraud data needed to make this decision is too sensitive to share across borders, teams, or with third parties.

### The Solution: ML on Encrypted Data

Instead of decrypting records, we train classifiers using only **aggregate counts**:

```
P(fraud_type = "mule_account" | high_risk) = count(mule_account AND high_risk) / count(high_risk)
```

These counts come from Blind Insight's encrypted indexes — the actual records are never decrypted. The same ~90 aggregate queries feed all three algorithms: Naive Bayes uses conditional probabilities, Decision Trees use Gini impurity gain, and Logistic Regression uses the feature covariance matrix (X'X) and target correlation (X'y).

**Why this works:** Blind Insight uses **two keys per field** — a one-way query key (for search and aggregation) and a separate field-level key (for encryption/decryption). Aggregate queries only require the query key. This means even a **Data Requester** — who has no field-level keys and *cannot decrypt a single record* — can train fully accurate ML models. The Data Owner controls who gets which keys, at which field level, all programmable via APIs. See [Key Sharing](https://docs.blindinsight.io/getting-started/key-sharing/) and [APPROACH.md](APPROACH.md) for details.

### Data Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                     YOUR LAPTOP (Jupyter)                        │
│  ┌─────────────────┐    ┌─────────────────┐    ┌──────────────┐ │
│  │  Notebook Cell  │───▶│  Python Client  │───▶│ Blind Proxy  │ │
│  │  (ML Training)  │    │  (HTTP Request) │    │ (local)      │ │
│  └─────────────────┘    └─────────────────┘    └──────┬───────┘ │
└────────────────────────────────────────────────────────┼────────┘
                                                         │
                                                         ▼
                                              ┌───────────────────┐
                                              │  Blind Insight    │
                                              │  (app.blind       │
                                              │   insight.io)     │
                                              │                   │
                                              │  Returns: COUNT   │
                                              │  (not raw data)   │
                                              └───────────────────┘
```

**Key point:** Only aggregate counts travel back. Individual records stay encrypted. The query key (a one-way hash) enables search and aggregation but **cannot decrypt data** - decryption requires field-level keys that only the Data Owner controls.

## Troubleshooting

### "Connection refused" or proxy errors

**Problem:** The Blind Insight proxy isn't running.

**Solution:**
```bash
./blind proxy
```
Keep this terminal open.

**Verification:**
```bash
curl -sk https://local.blindinsight.io/api/health/
```

### "Proxy auth not configured" or authentication errors

**Problem:** Blind Proxy isn't authenticated or credentials missing.

**Solution:**
1. Authenticate Blind Proxy:
   ```bash
   ./blind login
   ```
2. Ensure `.env` has your credentials:
   ```env
   BI_BACKEND=proxy
   BI_EMAIL=your-email@example.com
   BI_PASSWORD=your-password
   ```

**Verification:** Re-run the notebook setup cell; if auth is correct, warm-up will succeed and you'll see "Proxy warm-up" with timing info.

### Slow queries or timeouts

**Problem:** Blind Insight indexes are still building after upload.

**Solution:** Wait 30+ seconds after uploading data, then re-run the query. Server-side covering indexes make aggregate queries fast once indexing completes.

**Check indexing status:**
- Query should return counts > 0 if indexing is complete
- If counts are 0, wait longer and retry

### "Aggregate returns 0" or wrong counts

**Problem:** Query syntax incorrect, schema max values too low, or indexing incomplete.

**Solutions:**
1. **Check query syntax:** Use `risk_level:count(50~100)` NOT `risk_level:count(50, 100)`
2. **Verify schema max values:** Integer fields need `maximum = actual_max + 2` (see `schemas/fraud.json` for an example)
3. **Wait for indexing:** Wait 30+ seconds after upload before querying

### Wrong accuracy or predictions

**Problem:** Local SQLite data doesn't match Blind Insight data.

**Solution:** Regenerate local data with `python3 scripts/generate_fraud_data.py`. The script produces both SQLite databases and JSON upload batches from the same source, ensuring they match record-for-record.

### Database files missing

**Problem:** Demo requires these files but they're not present.

**Solution:** Run the generation script:
```bash
python3 scripts/generate_fraud_data.py
```
This creates both SQLite databases and JSON batch files.

### "Seed phrase" or keyring errors

**Problem:** Blind Proxy keyring corrupted or missing.

**Solution:** See [Create your Keyring](https://docs.blindinsight.io/getting-started/using-the-blind-proxy/#create-your-keyring) and [Troubleshooting](https://docs.blindinsight.io/getting-started/using-the-blind-proxy/#troubleshooting). Use `./blind keyring create` for a new keyring and `./blind keyring inspect` to verify credentials after `./blind login`.

### Import errors in notebook

**Problem:** Python dependencies not installed.

**Solution:**
```bash
pip install -r requirements.txt
```

**Verification:**
```bash
python -c "import pandas, sklearn, sqlite3; print('All imports OK')"
```

### Notebook cells hang or timeout

**Problem:** Too many parallel queries overwhelming the server.

**Solution:**
- The notebook uses batched parallel queries with ThreadPoolExecutor (3 batches, 30 workers)
- If you experience timeouts, check your network connection to the proxy
- Add delays between batches if needed

|-------|-----------|-------------|-----|-----------|---------------|
| **Naive Bayes** | 0.942 | 0.942 | 0pp | ~90 | Never |

| Aspect | Plaintext (sklearn) | Blind Insight | Overhead |
|--------|-------------------|--------------|----------|
| NB Training | ~0.01s | ~35s (local BI) / ~2min (cloud) | Network round-trips for ~90 queries |

## Technical Details

### Query Syntax

Blind Insight uses a simple filter syntax:

```python
# Count high-risk mule accounts
"risk_level:count(50~100),fraud_type:mule_account"

# Get average risk level
"risk_level:avg(0~100)"

# Filter by jurisdiction
"account_jurisdiction:us"
```

### Algorithms: Naive Bayes, Decision Trees, Logistic Regression

All three algorithms train from the same ~90 encrypted aggregate queries:

**Naive Bayes** — conditional probabilities from counts:
```
P(high_risk | features) ∝ P(high_risk) × ∏ P(feature_i | high_risk)
```

**Decision Trees** — Gini impurity gain from class counts per feature value (same criterion as sklearn's CART):
```
gini_gain = base_gini - weighted_sum(child_gini for each split)
```

**Logistic Regression** — OLS seed from X'X (marginal + pairwise counts) refined via IRLS (Newton-Raphson):
```
β = (X'X)⁻¹ X'y  →  refine with IRLS iterations  →  same MLE as sklearn
```

See [APPROACH.md](APPROACH.md) for detailed explanations of each algorithm and the full taxonomy of supported, approximately supported, and emerging algorithms.

**Contributions from the open-source community are very welcome** — if you build a demo for one of the [supported algorithms](APPROACH.md#what-algorithms-work-on-encrypted-data), please open a PR!

We'd love help expanding the library of ML-on-encrypted-data examples. Ideas for contributions:

- **New algorithm demos** — pick any algorithm from the [supported list in APPROACH.md](APPROACH.md#what-algorithms-work-on-encrypted-data) and build a notebook
- **New datasets / use cases** — healthcare, insurance, identity verification, etc.
- **Performance optimizations** — query batching, caching strategies, parallel execution
- **Documentation improvements** — clearer explanations, diagrams, translations

Please open an issue to discuss your idea before starting, then submit a PR. See [APPROACH.md](APPROACH.md) for technical details on which BI operations each algorithm needs.
